    Sandboxie注入

    Sandboxie 注入流程的一个整体梳理，帮助理解它是如何将自身的代码（SbieDll.dll）注入目标进程，并在进程启动或执行过程中完成一系列钩子和初始化操作的。

    1. 注入触发与初始流程

    1. 进程创建通知

      • Sandboxie 驱动通过 PsSetCreateProcessNotifyRoutine 回调获得进程创建的通知。
      • 如果判断该进程需要被沙箱化，就会阻塞创建流程，并让服务进程（SbieSvc）执行后续的注入（或者先创建挂起的进程，再通过驱动进行沙箱化注入）。

    2. 核心函数：SbieDll_InjectLow

      • 这是执行远程注入的主要函数：_FX ULONG SbieDll_InjectLow(HANDLE hProcess, BOOLEAN is_wow64, BOOLEAN bHostInject, BOOLEAN dup_drv_handle)
      • 其参数包括：目标进程句柄 hProcess、进程是否 WoW64（is_wow64）、是否以特殊模式注入（bHostInject）、以及是否需要将驱动句柄也一起传递（dup_drv_handle）等。

    3. 准备低级数据结构（SBIELOW_DATA）

      • 在注入之前，会先创建名为 lowdata 的数据结构（SBIELOW_DATA），并在其中存储进程/注入相关的各种标志和参数：
        • is_wow64
        • bHostInject
        • api_device_handle（可选）
        • … 等，用于后续在目标进程中使用。

    2. Shell Code（LowLevel.dll）的分配与写入

    1. 分配并写入 Shell Code

      • 调用 SbieDll_InjectLow_CopyCode 在目标进程中分配一段内存，保留给整个 shell code（LowLevel.dll）和它的跳转表（SBIELOW_J_TABLE），并将资源里的 shell code 内容直接写入到目标进程。
      • 然后还会拷贝一些 ntdll!LdrInitializeThunk 的前 48 个字节到 lowdata.LdrInitializeThunk_tramp（稍后会构造跳板用）。

    2. 可选：复制驱动句柄

      • 若 dup_drv_handle 参数为真，会调用 SbieDll_InjectLow_SendHandle 将驱动句柄复制给目标进程，并写入该句柄到 lowdata.api_device_handle。

    3. 保存各项地址 / 函数

      • 将进一步需要的 ntdll 函数地址、SBIELOW_J_TABLE 的地址等写入到注入数据块里 (lowdata 中)。

    4. 构造入口跳板

      • SbieDll_InjectLow_BuildTramp 在 LdrInitializeThunk_tramp 中建立一个跳板，用于将后续流程跳回到原先的 LdrInitializeThunk。

    5. 分配并写入 syscall 数据

      • SbieDll_InjectLow_CopySyscalls 再分配一段内存存放 “系统调用 hooking” 所需的元数据（syscall_data），并整合一些对 SBIELOW_EXTRA_DATA 的引用。
      • 其中包含了要注入的 SbieDll.dll 路径（32/64 位版本），以及部分加载信息。最后将 lowdata.syscall_data 与这块内存关联。

    6. 覆盖 ntdll!LdrInitializeThunk

      • 最后一步，调用 SbieDll_InjectLow_WriteJump 修改目标进程的 ntdll!LdrInitializeThunk（前几个字节）为跳转指令，指向写入进程内存中的 shell code 起始位置。

    结果：目标进程一旦恢复执行（Resume），它首先会执行被覆盖的 LdrInitializeThunk，从而跳进 shell code（LowLevel.dll）的入口点。

    3. Shell Code（LowLevel.dll）的执行

    1. Shell Code 入口：_Start

      • shell code 安装在进程内存后，代码默认基地址为 0（以形成位置无关）。
      • _Start 会计算出自己（shell code）在目标进程中的真实基地址，然后推算出写入的数据块指针（SBIELOW_DATA）和一批帮助函数的地址。
      • 最后它调用 EntrypointC 进入 C 语言部分。

    2. EntrypointC 主流程

      • 通过自旋锁防止重复执行。
      • 判断 data->bHostInject：
        • 若等于 0：调用 InitSyscalls 钩住所有 ntdll syscall，再调用 InitInject 做 DLL 注入准备，并在 64 位时进一步修改 ConsoleHandle。
        • 若不为 0：只调用 InitInject。
      • 最后，它跳转回原先保存在 data->LdrInitializeThunk_tramp（即最开始备份的 LdrInitializeThunk)。

    4. InitInject 与 RtlFindActivationContextSectionString 钩子

    1. 查找目标 ntdll 基址

      • 判断是原生 32 位/64 位进程，还是 wow64 环境。
      • 在 wow64 情况下，需要找到 32 位 ntdll 基址。Windows 8 之后不能直接使用 KUSER_SHARED_DATA::Wow64SharedInformation，故 Sandboxie 使用 NtQueryVirtualMemory 获取并保存在 ntdll_wow64_base 字段。

    2. 转换 syscall_data 内存为 INJECT_DATA

      • 在注入数据区（syscall_data）内部的一部分空间会临时用作 INJECT_DATA 结构，存放加载 DLL 所需的地址及路径、钩子信息等。

    3. 定位并钩住 RtlFindActivationContextSectionString

      • 解析 ntdll 导出表，找到 LdrLoadDll, LdrGetProcedureAddress, NtRaiseHardError 以及 RtlFindActivationContextSectionString 等地址。
      • 在 64 位或 32 位 ntdll 中，通过保存原函数开头若干字节，再使用跳转指令将 RtlFindActivationContextSectionString 的入口改写为 Sandboxie 的 detour 函数。

    4. RtlFindActivationContextSectionString Detour

      • 该 detour 会先还原 RtlFindActivationContextSectionString 的开头，然后加载 kernel32.dll、再加载 SbieDll.dll，并调用 SbieDll!Dll_Ordinal1（导出序号 1）。
      • 注意：因为这个 detour 在目标进程的同一位宽模式下执行（即 64 位进程为 64 位指令，wow64 进程为 32 位指令）。

    5. SbieDll.dll 的执行

    1. Dll_Ordinal1

      • 这是注入的主要入口函数；它读取传进来的 INJECT_DATA，获取 SBIELOW_DATA 的地址等关键信息。
      • 然后释放早先的那块 INJECT_DATA（不再需要）。
      • 依据 bHostInject:
        • 若 = 0：调用 SbieDll!Dll_InitInjected，执行各种 API 的 hooking，最后调用 SbieDll!Ldr_Init，它会设置 DLL 加载回调、再调用 SbieDll!Ldr_Inject_Init 等进行更多注入操作。
        • 若 != 0：直接调用 SbieDll!Ldr_Inject_Init。
      • 完成后，再执行原 RtlFindActivationContextSectionString（还原之前 detour 修改的参数），并返回给调用者。

    2. 进程主入口点的二次钩子

      • SbieDll!Ldr_Inject_Init 再次设置一个钩子，这次目标是可执行文件的真实 Entry Point：
        • 备份该入口点头几个字节，并改写为 jump 指令，引向 SbieDll!Ldr_Inject_Entry64 或 SbieDll!Ldr_Inject_Entry32。
      • 这样一来，目标进程运行到真正的入口点时，也会经过 SbieDll 代码进行最后阶段初始化。

    3. Ldr_Inject_Entry

      • 恢复原入口点指令，并让返回地址指向入口点开头（使后续仍能正常执行程序原代码）。
      • 如果 bHostInject = 0，在真正执行进程的入口点前，还会调用 SbieDll!Ldr_LoadInjectDlls 加载 Sandboxie.ini 配置的额外 DLL，然后再执行 SbieDll!Dll_InitExeEntry 做最后的注入初始化。
      • 若 bHostInject != 0，只执行 SbieDll!Ldr_LoadInjectDlls。
      • 完毕后返回到目标进程真正入口点，让应用程序继续正常运行。

    小结

    1. 主要组件

      • Loader/Injector：由服务或驱动调用 SbieDll_InjectLow，将 shell code（LowLevel.dll）及其配套数据写入目标进程，改写 ntdll!LdrInitializeThunk 以便在进程启动时接管流程。
      • Shell Code (LowLevel.dll)：进程开始执行后，首先进入该 shell code；它会根据场景完成提权、设置钩子等需求，然后跳回系统原生入口函数。
      • Payload (SbieDll.dll)：shell code 载入完 SbieDll.dll 并调用其导出函数 Dll_Ordinal1，这才是 Sandboxie 真正的“核心注入逻辑”，进一步对系统 API、进程入口点进行钩子或劫持，实现对目标进程的沙箱化控制。

    2. 多层“hooking”

      • 首先劫持 OS 的进程启动例程（LdrInitializeThunk），再劫持 RtlFindActivationContextSectionString 来加载自己的核心 DLL，随后还要在进程入口点加一层钩子，都旨在保证 Sandboxie 的注入逻辑在最早和最关键的时机介入。

    3. 32/64 位兼容

      • 在 64 位系统上针对 WoW64 进程，会在 shell code 或后续流程里专门定位 32 位 ntdll，并通过相应的调用和汇编跳转来实现注入。这使得 Sandboxie 可以同时处理原生 x64 和 WoW64 子系统的进程。

    整体而言，Sandboxie 的 DLL 注入流程高度依赖对底层 Windows 内核对象（进程、Section、ntdll 函数等）的操作以及对关键启动例程的修改，分层递进地实现对目标进程的控制和后续所有加载行为的监控或拦截。
